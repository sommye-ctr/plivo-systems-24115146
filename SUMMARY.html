Real-Time UDP Sender/Receiver

Architecture & design summary — protocol v3 (locked)

C++17UDPblock-parity FEC (K=2)forward-on-arrivaloverhead 1.54×grade @ delay_ms=120

Two C++ programs carry a 20 ms voice frame stream across a deliberately hostile UDP relay (drops, jitter, reordering, duplication). The goal: keep the deadline-miss rate under 1%, stay under 2× bandwidth, and add as little playout delay as possible.

1. System context & data path

Our code owns only the middle two boxes. The harness source, relay, and player are fixed and unmodifiable; the relay also meters every byte we send for the overhead cap.

  harness source                                             harness player
  (emits frame i     ┌─────────┐   hostile   ┌───────────┐   (judges frame i
   every 20ms)  ───▶ │ SENDER  │ ──relay──▶  │ RECEIVER  │ ─▶ against deadline
   :47010           │ :47001  │  drop/delay/ │  :47002   │    t0+delay+i·20ms)
                     └─────────┘  reorder/dup └───────────┘    :47020
                       our code                 our code
The scoring insight that shaped the whole design. The player (endpoints.py) marks frame i present iff any correct-seq packet for it arrives before t0 + delay_ms + i·20ms. It dedups by seq itself and never penalizes early arrival. So the receiver's job is simply to get each correct frame to the player as early as possible — there is nothing to be gained by buffering on our side.

2. Wire protocol

One flat packet shape for everything — no variable-length forms to special-case.

  ┌───────────────── 4 bytes ─────────────────┬──────── 160 bytes ────────┐
  │ bit 31 = is_parity                         │                           │
  │ bits 0..30 = seq (data) / block_id (parity)│  payload (opaque, random) │
  └────────────────────────────────────────────┴───────────────────────────┘
  164 bytes, big-endian header. Data packet: is_parity=0, carries frame `seq`.
  Parity packet: is_parity=1, carries XOR of the K frames in `block_id`.

Payloads are SHA-256-derived and incompressible, so every redundant byte costs its full 160 bytes — there is no compression trick, only how cleverly we spend the byte budget.

3. Sender architecture

Single-threaded and stateless beyond one running XOR accumulator. The harness already paces us at 20 ms, so we never batch or add delay.

  • On each frame: forward the data packet immediately (header = seq).
  • Maintain block_xor = XOR of the K=2 frames in the current block; reset it at the block's first frame (seq % K == 0), fold in each subsequent frame.
  • At the last frame of a block (seq % K == K-1): emit one extra parity packet (header = block_id | 0x80000000, payload = block_xor).

Net result: for every K data packets we add exactly one parity packet, so bandwidth is a fixed (K+1)/K × 164/160 = 1.54× — independent of the network, hence identical on any grading profile.

4. Receiver architecture

A single receive loop. No playout thread, no clock, no mutex — the simplification is the design.

4.1 Forward-on-arrival

Every data frame is forwarded to the player the moment it arrives; every reconstructed frame is forwarded the moment it is rebuilt. Because the player enforces the deadline and dedups, this is both correct and optimal: a frame can only help by arriving earlier. A best-effort seq dedup (sent_seq[seq % window]) avoids duplicate sends and wasted reconstruction work.

4.2 Block reconstruction (single-erasure FEC)

Per block we keep a BlockState: a got[K] bitmap, a running partial_xor of the data frames seen, and the parity payload once it arrives. On every arrival we update the block and call try_reconstruct: if the parity is in and exactly K-1 of the K data frames are in, the missing frame is partial_xor ⊕ parity — rebuilt and forwarded through the same path as a normal arrival.

4.3 Retirement horizon (bounded memory, safe slot reuse)

Blocks live in a ring indexed by block_id % block_capacity. A frame's deadline is a known function of its seq, so once we have seen a seq window ahead, older frames are certainly expired. floor_seq = max_seq_seen − window retires them, so a late or duplicate packet for an expired block can never clobber a slot now reused by a newer block. Memory is bounded to the playout horizon, not the stream length.

5. Design choices & trade-offs

FEC, not retransmission

A NACK would have to cross the hostile relay twice (request + resend), each leg paying the full 10–80 ms delay range and its own loss risk — 20–160 ms added to the exact quantity we are scored on. Proactive parity turns "did this frame survive?" into "did the frame and its independently-routed parity both get unlucky?", recovered with zero round trip.

Block parity at K=2, not larger K and not full duplication

Recovery latency is ~one frame (20 ms): parity for a block is available right after the block's last frame. K=4 (1.28× overhead) recovers too little — a 5%-loss profile too often takes a second hit inside one wide block. Full duplication (~2.0×) leaves no headroom. K=2 is the sweet spot: enough recovery strength for a 5% profile at a comfortable 1.54× with real margin under the cap.

No receiver playout clock — the decisive fix

Earlier versions ran a sleep_until jitter buffer that held each frame to ~its deadline before emitting. Since the player only cares about arrival-before-deadline, that buffering bought nothing and its wake-ups were subject to OS scheduling jitter — which made identical-seed runs on profile B straddle the 1% cap (1.07% / 1.00% / 1.00%). Removing it made the same runs stable at 0.60%.

Delay budget

The floor is intrinsic to the profile: B's ~80 ms jitter tail + ~20 ms parity lag ⇒ ~100 ms. We grade at 120 ms — a deliberate margin over the floor, because grading profiles are unseen and may be harsher than B.

7. Known failure modes

  • 2-loss bursts within a block. Single parity recovers one erasure per block; losing both data frames of a block is unrecoverable. Practice profiles A/B are i.i.d. and don't exercise this, but the relay supports Gilbert-Elliott burst loss.
  • Jitter tail beyond the chosen delay. A frame delayed past t0 + delay_ms + i·20ms is an unavoidable miss regardless of FEC; a profile with a >100 ms tail would need a larger delay_ms.
  • Mitigation not taken: interleaving block members (pairing frames a stride apart) buys burst tolerance but adds ~stride×20 ms of delay — a bad trade against the primary score for a hypothetical hazard, so we left it out deliberately.
